## Attacker Path

**preconditions:**
- `liquidityAddedBuild == true` (confirmed live: `true`)
- A non-zero `titanXIntervalsBuild[lastTitanXBuildIntervalNumber].amountAllocated` exists and `amountBurned == 0` (i.e., the current interval has not been consumed)
- Attacker has sufficient TitanX capital to move the TitanX/TORUS pool spot price by up to ~10% relative to the 15-minute TWAP

**attacker-controlled inputs:**
- Front-run tx: buy Torus with TitanX on the TitanX/TORUS pool, pushing spot price up (Torus becomes more expensive in TitanX terms)
- Victim tx: `swapTitanXForTorusAndBuild(_deadline)` — any unprivileged caller
- Back-run tx: sell Torus back for TitanX at the now-higher price

**call sequence:**
1. Attacker front-runs: large TitanX→Torus swap on the pool, moving spot price up by ≤10% vs TWAP
2. `swapTitanXForTorusAndBuild` executes:
   - `_swapTitanXForTorus(half, _deadline)`: TWAP oracle returns pre-manipulation quote; `amountOutMinimum = TWAPquote * 90%`; actual swap executes at manipulated spot price, returning fewer Torus (between 90–100% of TWAP quote) — swap does not revert
   - `_addLiquidityBuild(remainder, torusPurchased, _deadline)`: `remainder` TitanX + fewer-than-expected Torus passed; `increaseLiquidity` uses tokens in the ratio of the current (manipulated) pool price; since Torus is now more expensive, the LP absorbs less TitanX per unit of Torus; leftover TitanX (`amount0 - used0` or `amount1 - used1`) is transferred to `genesisWallet`
3. Attacker back-runs: sells Torus for TitanX at the elevated price, capturing sandwich profit [4](#0-3) [5](#0-4) 

## Why Existing Checks Fail

**TWAP oracle (`getTorusQuoteForTitanX`):** Uses a 15-minute TWAP window. A same-block front-run does not move the TWAP, so `amountOutMinimum` is computed from the pre-manipulation price. The attacker can move the spot price up to 10% below the TWAP quote without triggering a revert. The oracle provides no protection against within-tolerance spot manipulation. [6](#0-5) 

**`liquiditySlippage` (10%):** `amount0Min` and `amount1Min` in `_addLiquidityBuild` are 90% of desired. This means up to 10% of each token can go unused and be sent to `genesisWallet`. The attacker's manipulation is designed to stay within this tolerance. [7](#0-6) 

**`burnTorus()` at end of `swapTitanXForTorusAndBuild`:** Called after `_addLiquidityBuild` returns. Any leftover Torus has already been transferred to `genesisWallet` inside `_addLiquidityBuild` (line 1014), so `burnTorus()` burns 0 or near-0. The leftover TitanX is also already gone to `genesisWallet` (line 1011). Neither check recovers the diverted funds. [8](#0-7) 

**`nonReentrant`:** Prevents reentrancy but does not prevent sandwich MEV across separate transactions.

## Rejection Checks

**expected behavior checked:** The `_addLiquidityBuild` NatSpec says "refunding any slippage leftovers" — the design intent is that leftovers are dust. However, the 10% slippage tolerance is wide enough for a deliberate attacker to make leftovers non-trivial. This is not purely expected behavior at scale. [9](#0-8) 

**prior report checked:** No prior report found in the indexed codebase.

**README/NatSpec checked:** No NatSpec or README language explicitly acknowledges sandwich risk or designates `genesisWallet` transfers as acceptable for large amounts.

**unsupported assumption checked:** The attack does not require oracle failure, malicious token owner action, or impossible values. It requires only normal AMM price manipulation within the declared slippage tolerance, which is a standard and reachable MEV technique on Ethereum mainnet.

    function getTorusQuoteForTitanX(
        uint256 baseAmount
    ) public view returns (uint256 quote) {
        address poolAddress = torusToken.titanXTorusPool();
        uint32 secondsAgo = 15 * 60;
        uint32 oldestObservation = OracleLibrary.getOldestObservationSecondsAgo(
            poolAddress
        );
        if (oldestObservation < secondsAgo) secondsAgo = oldestObservation;
        (int24 arithmeticMeanTick, ) = OracleLibrary.consult(
            poolAddress,
            secondsAgo
        );
        uint160 sqrtPriceX96 = TickMath.getSqrtRatioAtTick(arithmeticMeanTick);
        quote = OracleLibrary.getQuoteForSqrtRatioX96(
            sqrtPriceX96,
            baseAmount,
            TITAN_X_ADDRESS,
            address(torusToken)
        );
    }
