## Attacker Path

**preconditions:**
- Pool (`torusToken.titanXTorusPool()`) has Uniswap V3 observation cardinality = 1 (default, never increased via `increaseObservationCardinalityNext`)
- Pool has low swap activity so that the last observation timestamp is recent (e.g., 12 seconds ago = 1 block)
- `liquidityAddedBuild = true` and a non-zero `titanXIntervalsBuild` allocation exists for the current interval

**attacker-controlled inputs:**
- Block N: attacker executes a large swap in the TORUS/TITAN_X pool, pushing the tick to a very low Torus price (Torus appears cheap in TitanX terms). This writes a new observation with `observationTimestamp = block.timestamp(N)`.
- Block N+1: attacker calls `swapTitanXForTorusAndBuild(deadline)`.

**call sequence:**
1. Block N: `attacker → UniswapV3Pool.swap(...)` — manipulates tick, writes observation at `T_N`
2. Block N+1 (same tx or front-run): `attacker → swapTitanXForTorusAndBuild(deadline)`
   - `getOldestObservationSecondsAgo` returns `block.timestamp(N+1) - T_N ≈ 12`
   - `secondsAgo = min(900, 12) = 12` — TWAP window collapses to 12 seconds
   - `getTorusQuoteForTitanX` returns a near-zero quote (manipulated tick)
   - `amountOutMinimum = near_zero * 0.9 ≈ 0`
   - Uniswap router executes the swap with effectively no slippage protection
3. Attacker (as MEV sandwich) front-runs the actual swap in block N+1 to extract the price impact [3](#0-2) [4](#0-3) 

## Why Existing Checks Fail

**TWAP window fallback is the root cause.** The 15-minute TWAP is only used when `oldestObservation >= 900`. With cardinality = 1 (Uniswap V3 default), the pool stores exactly one observation slot. `getOldestObservationSecondsAgo` returns `block.timestamp - lastObservationTimestamp`, which equals the time since the last swap. If the attacker's manipulation trade was the most recent swap, this is ~12 seconds (1 block), collapsing the TWAP window to 12 seconds. [5](#0-4) 

**`titanXToTorusSlippage = 10` is applied to the manipulated quote, not the fair price.** With a 10x price manipulation, `amountOutMinimum = fair_value * 0.1 * 0.9 = 9% of fair value`. The attacker can extract up to ~91% of the Torus value. [6](#0-5) 

**`nonReentrant` does not help** — the attack is a cross-block sandwich, not a reentrancy.

**`secondsAgo = 0` revert is not a defense** — the attacker manipulates in block N, not block N+1, so `oldestObservation ≥ 12 > 0`. [7](#0-6) 

## Rejection Checks

**expected behavior checked:** No — the protocol explicitly uses a TWAP to prevent sandwich attacks. The fallback to a 12-second window defeats this intent entirely.

**prior report checked:** Not found in the repository.

**README/NatSpec checked:** No NatSpec on `getTorusQuoteForTitanX` documents the cardinality-dependent window collapse as intentional.

**unsupported assumption checked:** The attack does not require oracle failure or malicious token owner. It requires only that the pool's observation cardinality is 1 (the Uniswap V3 default) and that the attacker can be the last trader before the victim's block. Both are realistic on mainnet.         uint32 _deadline
    ) external nonReentrant buildTitanXIntervalUpdate{
        if (!liquidityAddedBuild) revert NotStartedYet();

        IntervalBuild storage currInterval = titanXIntervalsBuild[
            lastTitanXBuildIntervalNumber
        ];
        if (currInterval.amountBurned != 0) revert IntervalAlreadyUsed();

        currInterval.amountBurned = currInterval.amountAllocated;
        uint256 incentive = (currInterval.amountAllocated *
            INCENTIVE_FEE_BUILD) / BPS_DENOM;

        uint256 titanXToUse = currInterval.amountAllocated - incentive;
        uint256 half = titanXToUse / 2;
        uint256 remainder = titanXToUse - half;

        totalTitanXForBuild -= titanXToUse + incentive;
        titanXUsedForBuilds += titanXToUse;

            releaseFractalFunds();
