## Attacker Path

**preconditions:**
- The Torus/TitanX Uniswap V3 pool was created by `addLiquidityToTorusTitanXPool` with no subsequent call to `increaseObservationCardinalityNext`. The default cardinality is 1, so the pool stores only one observation slot. With cardinality=1, `getOldestObservationSecondsAgo` reads `observations(0)` (the only slot), whose `observationTimestamp` is the timestamp of the last block that touched the pool. If the pool was touched in the previous block, `oldestObservation ≈ 12 s`, far below the 15-minute target.
- `liquidityAdded == true` (confirmed live: `true`).
- `titanXToTorusSlippage == 10` (confirmed live: 10%), giving a 10% tolerance band.
- The current interval has not yet been burned (`currInterval.amountBurned == 0`).

**attacker-controlled inputs:**
- Block N: attacker executes a large buy of Torus in the Torus/TitanX pool (pushes spot price of Torus up, i.e., fewer Torus per TitanX). This sets the single stored observation to the manipulated tick.
- Block N+1: attacker submits a Flashbots bundle:
  1. Tx A: buy Torus again (front-run, push price further up).
  2. Tx B: call `swapTitanXForTorusAndBurn(deadline)` — the TWAP window is ~12 s, reflecting the block-N manipulation; `amountOutMinimum` is set low.
  3. Tx C: sell Torus (back-run, capture profit).

**call sequence:**
```
Block N:   attacker.buyTorus(largeAmount)          // manipulate pool price
Block N+1 bundle:
  Tx A:    attacker.buyTorus(amount)               // front-run
  Tx B:    TorusBuyAndProcess.swapTitanXForTorusAndBurn(deadline)
             → _intervalUpdateTitanXForBurning()
             → _swapTitanXForTorus(titanXToSwapAndBurn, deadline)
                 → getTorusQuoteForTitanX()
                     oldestObservation ≈ 12 s < 900 s → secondsAgo = 12
                     TWAP reflects block-N manipulated (high Torus price)
                     expectedTorusAmount is low
                     amountOutMinimum = expectedTorusAmount * 90 / 100  (very low)
                 → ISwapRouter.exactInput(params)  // executes at inflated price, passes low minimum
  Tx C:    attacker.sellTorus(amount)              // back-run, extract profit
```

## Why Existing Checks Fail

1. **TWAP window collapses to `oldestObservation`**: The guard `if (oldestObservation < secondsAgo) secondsAgo = oldestObservation` is intended as a safety fallback for young pools, but it silently degrades the oracle to a near-spot price when cardinality=1. There is no minimum floor on `secondsAgo` other than `!= 0`. [1](#0-0) 

2. **No cardinality increase**: The protocol never calls `increaseObservationCardinalityNext` on the Torus/TitanX pool. With cardinality=1, a single block's manipulation fully controls the TWAP.

3. **10% slippage tolerance is insufficient**: With a 12-second TWAP window, the attacker can move the price by more than 10% in one block on a low-liquidity pool (initial liquidity: 200B TitanX / 20K Torus). The `amountOutMinimum` can be set to 81% of fair value (90% of a TWAP that is already 10% below fair), allowing the sandwich to extract up to ~19% of the swap. [5](#0-4) 

4. **`consult` accepts any nonzero `secondsAgo`**: The only guard is `require(secondsAgo != 0, "BP")`. A 12-second window passes this check and produces a manipulable TWAP. [3](#0-2) 

5. **`swapTitanXForTorusAndBurn` is permissionless**: Any caller (including the attacker) can trigger the swap, giving the attacker full control over timing. [6](#0-5) 

## Rejection Checks

**expected behavior checked:** The fallback to `oldestObservation` is a deliberate design choice for young pools, but it creates a manipulable oracle when cardinality=1. This is not documented as an accepted risk.

**prior report checked:** No evidence of a prior report in the repository.

**README/NatSpec checked:** No NatSpec on `getTorusQuoteForTitanX` acknowledges the cardinality=1 / short-window risk.

**unsupported assumption checked:** The attack does not require impossible oracle values or malicious token owner action. It requires only that the attacker have sufficient capital to move the price in a low-liquidity pool for one block, which is realistic given the pool's initial liquidity parameters (`INITIAL_TITAN_X_FOR_LIQ = 200_000_000_000e18`, `INITIAL_LP_MINT = 20_000e18`). [7](#0-6)
