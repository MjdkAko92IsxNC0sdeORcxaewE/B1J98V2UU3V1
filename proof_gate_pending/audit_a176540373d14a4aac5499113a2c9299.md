## Attacker Path

**preconditions:**
- Attacker has capital sufficient to move the TITAN_X_ETH_POOL TWAP over the observation window (up to 15 minutes / 900 seconds). The actual cost depends on pool liquidity depth.
- A new ETH burn interval is ready to be triggered (i.e., `currInterval.amountBurned == 0` and `timeElapsed > INTERVAL_TIME`).

**attacker-controlled inputs:**
- Sustained buy-side pressure on `TITAN_X_ETH_POOL` over the TWAP window, pushing the tick cumulative upward (making TitanX appear more expensive in ETH terms, i.e., fewer TitanX per ETH in the TWAP quote).
- Front-run transaction buying TitanX immediately before the victim `swapETHForTorusAndBurn` call.
- Back-run transaction selling TitanX immediately after.

**call sequence:**
1. Attacker buys TitanX on `TITAN_X_ETH_POOL` continuously for ≥ `secondsAgo` seconds, shifting the TWAP tick upward (fewer TitanX per ETH).
2. `getTitanXQuoteForETH(ethToSwapAndBurn)` now returns a deflated value.
3. `adjustedTitanXAmount = deflatedQuote * 90 / 100` — `amountOutMinimum` is now well below fair market.
4. Attacker front-runs `swapETHForTorusAndBurn` with a large TitanX buy, pushing spot price further.
5. Protocol's `exactInput` swap executes at the inflated spot price; the low `amountOutMinimum` does not revert it.
6. Attacker back-runs by selling TitanX, capturing the spread.

## Why Existing Checks Fail

The TWAP is the only slippage guard for the ETH→TitanX leg. The 10% `ethToTitanXSlippage` tolerance is applied **on top of** the TWAP quote:

```solidity
uint256 adjustedTitanXAmount = (expectedTitanXAmount * (100 - ethToTitanXSlippage)) / 100;
``` [4](#0-3) 

If the TWAP itself is deflated by X%, the effective floor is `(1 - X%) * 90%` of fair value. A 10% TWAP deflation combined with the 10% slippage tolerance gives the attacker a ~19% window to execute a sandwich without reverting. The `oldestObservation` fallback can further shorten the effective TWAP window if pool cardinality is low, making manipulation cheaper:

```solidity
if (oldestObservation < secondsAgo) secondsAgo = oldestObservation;
``` [5](#0-4) 

There is no secondary price check, no circuit breaker, and no deviation bound between TWAP and spot.

## Rejection Checks

**expected behavior checked:** TWAP-based slippage is the intended protection mechanism, but the design assumes the TWAP itself is manipulation-resistant. For a pool with moderate liquidity, this assumption can fail.

**prior report checked:** No evidence of a prior report in the repository for this specific path.

**README/NatSpec checked:** No NatSpec on `getTitanXQuoteForETH` or `_swapETHForTitanX` documents TWAP manipulation as an accepted risk.

**unsupported assumption checked:** The attack requires the attacker to profitably manipulate the TWAP. This is capital-intensive and may not be profitable against a high-liquidity pool. The feasibility is the key unknown that requires a fork test to resolve.

        uint256 ethAmount,

                deadline: _deadline,
                amountIn: ethAmount,

    function getTitanXQuoteForETH(
        uint256 baseAmount
    ) public view returns (uint256 quote) {
        address poolAddress = TITAN_X_ETH_POOL;
        uint32 secondsAgo = 15 * 60;
        uint32 oldestObservation = OracleLibrary.getOldestObservationSecondsAgo(
            poolAddress
        );
        if (oldestObservation < secondsAgo) {
            secondsAgo = oldestObservation;
        }
        (int24 arithmeticMeanTick, ) = OracleLibrary.consult(
            poolAddress,
            secondsAgo
        );
        uint160 sqrtPriceX96 = TickMath.getSqrtRatioAtTick(arithmeticMeanTick);
        quote = OracleLibrary.getQuoteForSqrtRatioX96(
            sqrtPriceX96,
            baseAmount,
            WETH_ADDRESS,
            TITAN_X_ADDRESS
        );
    }
```
