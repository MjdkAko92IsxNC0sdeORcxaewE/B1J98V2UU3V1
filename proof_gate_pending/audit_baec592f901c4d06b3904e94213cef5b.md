## Attacker Path

**preconditions:**
- TitanX/TORUS pool has default Uniswap V3 observation cardinality of 1 (protocol never calls `increaseObservationCardinalityNext`). With cardinality 1, `getOldestObservationSecondsAgo` returns `block.timestamp - observations(0).timestamp`, which equals the time since the last swap — potentially seconds. This collapses the TWAP window to near-zero, making the oracle equivalent to spot price.
- `ethToTitanXSlippage = 10` and `titanXToTorusSlippage = 10` (live confirmed values).
- `TITAN_X_ETH_POOL` is an established pool; TWAP manipulation there is expensive but the TitanX/TORUS pool is protocol-created with likely cardinality 1.

**attacker-controlled inputs:**
- Attacker makes a swap on the TitanX/TORUS pool immediately before the victim tx to reset the observation timestamp to the current block, collapsing `secondsAgo` to near-zero.
- Attacker front-runs `swapETHForTorusAndBurn` with a large TitanX→TORUS buy (moving spot price up), then back-runs with a sell.

**call sequence:**
1. Attacker swaps on TitanX/TORUS pool to reset observation timestamp (collapses TWAP window).
2. Attacker front-runs: large buy on TitanX/TORUS pool, moving spot price up.
3. `swapETHForTorusAndBurn` executes: `getTorusQuoteForTitanX` reads the manipulated near-spot TWAP, producing an inflated `expectedTorusAmount`. `amountOutMinimum` = 90% of this inflated value — but the actual pool price after the front-run is even higher, so the swap still executes and the attacker's back-run captures the difference.

Wait — the direction matters. For a sandwich to work, the attacker front-runs to move price *against* the victim (buy before victim buys → price goes up → victim gets fewer tokens → attacker sells at higher price). The `amountOutMinimum` must not block this. If the TWAP is collapsed to spot, the `amountOutMinimum` tracks the manipulated spot, so it does not protect the victim. [6](#0-5) 

## Why Existing Checks Fail

**`oldestObservation` fallback is the root cause.** Both oracle functions cap `secondsAgo` at `oldestObservation`:

```solidity
if (oldestObservation < secondsAgo) secondsAgo = oldestObservation;
```

With Uniswap V3 default cardinality of 1, `getOldestObservationSecondsAgo` computes `block.timestamp - observations(0).timestamp`. Since cardinality is 1, `(observationIndex + 1) % 1 = 0`, so it reads the most recent observation. If the attacker (or any trader) made a swap in the same block, `secondsAgo` = 0, which causes `consult` to revert on `require(secondsAgo != 0, "BP")`. But if the attacker's swap was 1–2 blocks ago, `secondsAgo` = 12–24 seconds — far shorter than the intended 15-minute window.

The protocol never expands observation cardinality on the TitanX/TORUS pool it creates, so this fallback is always active for that pool. [7](#0-6) [6](#0-5) 

**The "cascade" claim is partially inaccurate.** Manipulating leg 1 (ETH→TitanX) does not weaken the *percentage* protection on leg 2. The second leg's `amountOutMinimum` is computed from the actual TitanX received, so it scales proportionally. The two legs are independently sandwichable, not multiplicatively compounding. The real vulnerability is on the TitanX/TORUS leg alone, due to low cardinality.

**Slippage tolerance is 10% (owner-set, not hardcoded).** The live value is confirmed at 10% for both legs. This is the maximum extractable per leg per sandwich. [8](#0-7) 

## Rejection Checks

**expected behavior checked:** The `oldestObservation` fallback is a deliberate safety measure to avoid reverting on new pools, but it creates the oracle weakness. This is not documented as an accepted risk.

**prior report checked:** No evidence of a prior report in the repository.

**README/NatSpec checked:** No NatSpec on `getTorusQuoteForTitanX` or `getTitanXQuoteForETH` acknowledges the cardinality/window risk.

**unsupported assumption checked:** The claim requires cardinality 1 on the TitanX/TORUS pool. This is the default in Uniswap V3 and the protocol code contains no call to `increaseObservationCardinalityNext`. This must be verified on-chain via a fork test.

**test setup:**

**expected assertion:**
- Attacker's net profit in TitanX/ETH terms > 0 after accounting for gas and swap fees.

- Compare attacker profit vs. a baseline run with a 15-minute TWAP (expanded cardinality) to show the cardinality-1 condition is the enabling factor.

**failure condition:** If `observationCardinality > 1` on the TitanX/TORUS pool, the TWAP window is longer and manipulation cost increases significantly — reject the finding. If the attacker's net profit is negative after fees, reject.
