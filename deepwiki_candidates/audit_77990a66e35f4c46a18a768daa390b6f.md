- No `swapETHForTorusAndBuild` calls for ≥ 30 days (passive condition; attacker just waits or exploits an existing gap)

**attacker-controlled inputs:**

**call sequence:**
1. Warp/wait ≥ 30 days since `lastETHBuildIntervalStartTimestamp`
2. Call `swapETHForTorusAndBuild(_deadline)` as any EOA

**Arithmetic trace:**
- `timeElapsed` ≥ 30 days = 4320 intervals
- `amountPerInterval = (totalETHForBuild * 1000 / 10000) / 144 = totalETHForBuild / 1440`
- `additionalAmount = amountPerInterval * (4320 - 1) ≈ 3 * totalETHForBuild`
- `_totalAmountForInterval` overflows `totalETHForBuild` → **capped at `totalETHForBuild`**
- `currInterval.amountAllocated = totalETHForBuild`
- `incentive = totalETHForBuild * 150 / 10000` = **1.5% of entire ETH build pool**
- `(bool success,) = payable(msg.sender).call{value: incentive}("")` → attacker receives ETH

## Why Existing Checks Fail

The `IntervalAlreadyUsed` guard (`currInterval.amountBurned != 0`) only prevents replay within the same interval number — it does not prevent the accumulation exploit. [6](#0-5) 

The cap `if (_totalAmountForInterval > totalETHForBuild)` is intended as a safety bound but is the root cause: it sets `amountAllocated` to the full pool balance, making the 1.5% incentive apply to the entire pool rather than a single interval's allocation. [1](#0-0) 

The `totalETHForBuild -= ethToSwapAndBuild + incentive` correctly drains the full pool in one call, confirming the accounting is consistent with the attacker receiving the oversized incentive. [7](#0-6) 

## Rejection Checks

**expected behavior checked:** The design intent is that each 10-minute interval releases a small fraction (10%/144 ≈ 0.069%) of the pool. Accumulating missed intervals to drain 100% of the pool in one call and paying 1.5% of the entire pool as incentive is not consistent with the per-interval incentive design. The cap is a guard against over-allocation, not a mechanism to enable full-pool single-call drains.

**prior report checked:** No NatSpec, README, or known-issues documentation in the repository describes this as intentional behavior.

**README/NatSpec checked:** No NatSpec on `_calculateIntervalsETHBuild` or `swapETHForTorusAndBuild` documents the cap-to-full-pool behavior as expected.

**unsupported assumption checked:** The precondition (30+ days of no build calls) is realistic — live state shows `lastETHBuildIntervalStartTimestamp = 1784012400` and `lastETHBuildIntervalNumber = 53070`, and the live event sample shows zero `BuyAndBuild` events in the last 10,000 blocks, confirming low activity.

**Live impact bound:** `totalETHForBuild = 106634700085918065` wei → max incentive = `106634700085918065 * 150 / 10000 ≈ 1,599,520,501,288,770` wei ≈ **0.0016 ETH** at current live state. Impact scales with pool size; if `totalETHForBuild` grows (e.g., after large `distributeETHForBuilding` deposits), the extractable incentive grows proportionally.
