## Attacker Path

**preconditions:**
- The TORUS/TITAN_X Uniswap V3 pool has `observationCardinality == 1` (the default for a newly created pool; no one has called `increaseObservationCardinalityNext`). This is the pool created by `addLiquidityToTorusTitanXPool`.
- The last swap in the pool occurred in the previous block (≤12 seconds ago), so `getOldestObservationSecondsAgo` returns ≤12.
- `liquidityAdded == true` and the current interval has `amountBurned == 0` (i.e., the interval is ready to be burned).

**attacker-controlled inputs:**
- A large TORUS sell into the pool in the same block (front-run), depressing the TORUS/TITAN_X spot price.
- A call to `swapTitanXForTorusAndBurn(deadline)` as the victim transaction.
- A TORUS buy-back after the victim transaction (back-run).

**call sequence:**
1. Attacker sells TORUS → TITAN_X in the TORUS/TITAN_X pool (depresses TORUS price, raises TITAN_X price in pool terms).
2. `swapTitanXForTorusAndBurn(deadline)` is called (by anyone — permissionless).
3. Inside: `getTorusQuoteForTitanX(titanXToSwapAndBurn)` is called.
4. `getOldestObservationSecondsAgo` returns `N` seconds (≤12 for a pool with cardinality==1 and recent activity).
5. `secondsAgo` is clamped to `N` (instead of 900).
6. `consult(pool, N)` computes a TWAP over `N` seconds — essentially the manipulated spot price.
7. `expectedTorusAmount` is low (reflects the depressed TORUS price).
8. `amountOutMinimum = expectedTorusAmount * 90 / 100` is set near-zero relative to fair value.
9. The Uniswap router executes the swap at the depressed price; the protocol receives far fewer TORUS than fair value.
10. Attacker buys TORUS back at the depressed price, profiting from the price impact of the protocol's large TITAN_X → TORUS swap. [4](#0-3) [5](#0-4) 

## Why Existing Checks Fail

The `titanXToTorusSlippage` guard (default 10%, line 829) is applied as a percentage of the oracle quote, not of the true fair price. When the oracle quote itself is derived from a manipulated near-spot TWAP, the 10% tolerance is 10% of the manipulated price — not 10% of the fair market price. The guard is therefore ineffective against oracle manipulation.

The `require(secondsAgo != 0, "BP")` in `consult` only prevents a zero-second window; it does not prevent a 1–12 second window, which is still fully manipulable within a single block.

The clamping logic `if (oldestObservation < secondsAgo) secondsAgo = oldestObservation` is the root cause: it silently degrades the TWAP protection from 15 minutes to near-spot whenever the pool's observation history is shallow (cardinality==1). [6](#0-5) [7](#0-6) [8](#0-7)
