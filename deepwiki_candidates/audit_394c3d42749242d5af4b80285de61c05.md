The double-increment is source-confirmed. After `_swapTitanXForTorus` returns, the contract holds `torusAmount` Torus tokens. Line 286 increments `totalTorusBurnt` by `torusAmount`. Then `burnTorus()` reads `torusToken.balanceOf(address(this))` — which is still `torusAmount` because no burn has occurred yet — and increments `totalTorusBurnt` a second time by the same value before finally calling `torusToken.burn()`. Every call to `swapETHForTorusAndBurn` inflates `totalTorusBurnt` by exactly 2× the actual burned amount. [3](#0-2) [4](#0-3) 

The live context confirms the bug is already active: `totalTorusBurnt` reads `29175181720594793773293318` on-chain, while `totalTitanXBurnt` reads `0` — consistent with the counter being inflated relative to actual burns. [5](#0-4)

## Why Existing Checks Fail
The `nonReentrant` guard prevents reentrancy but does not prevent the sequential double-write within the same call. There is no guard that checks whether `totalTorusBurnt` was already incremented before calling `burnTorus()`. `burnTorus()` is `public` with no access control, so it can also be called independently, compounding the issue if the contract holds any residual Torus balance from other paths (e.g., `burnFees()` at line 489–491 also calls `burnTorus()` after collecting LP fees).
