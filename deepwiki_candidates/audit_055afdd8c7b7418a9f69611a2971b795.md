## Attacker Path

**preconditions:**
- `liquidityAdded == true` (confirmed live: `true`)
- Sufficient time has elapsed since `lastBurnedETHIntervalStartTimestamp` such that `missedIntervals` is large enough for `amountPerInterval * (missedIntervals + 1) >= totalETHBurn`
- With live `DAILY_ALLOCATION_ETH_BURNING = 500` bps (5%), `INTERVALS_PER_DAY = 144`: `amountPerInterval = totalETHBurn * 500 / 10000 / 144`. The cap is hit when `missedIntervals + 1 >= 10000/500 * 144 = 2880` intervals = **20 days** of inactivity
- `currInterval.amountBurned == 0` for the new interval (guaranteed on first call after gap)

**attacker-controlled inputs:**
- `_deadline`: any valid future timestamp
- Timing: attacker waits until 20+ days have elapsed since the last ETH burn interval was processed

**call sequence:**
1. Wait 20+ days since last `swapETHForTorusAndBurn` call
2. Call `swapETHForTorusAndBurn(_deadline)` as any EOA

**Execution trace:**
- `burnETHIntervalUpdate` modifier calls `_intervalUpdateETHForBurning()`
- `timeElapsed > INTERVAL_TIME` → calls `_calculateIntervalsETHBurn(timeElapsed)`
- `amountPerInterval * (missedIntervals + 1) >= totalETHBurn` → cap fires: `_totalAmountForInterval = uint128(totalETHBurn)`
- `ethIntervalsBurn[_lastInterval].amountAllocated = totalETHBurn`
- Back in `swapETHForTorusAndBurn`: `amountAllocated = totalETHBurn`
- `incentive = totalETHBurn * 150 / 10000` (1.5% of entire pool)
- ETH sent to `msg.sender` via `.call{value: incentive}`
- `totalETHBurn -= totalETHBurn` → pool zeroed

## Why Existing Checks Fail

The only guard is the cap itself: [2](#0-1) 

This cap is intended to prevent over-allocation but instead **enables** the attack: it sets `amountAllocated` to the full pool balance, making the 1.5% incentive apply to the entire pool rather than a single interval's share. The `IntervalAlreadyBurned` check only prevents replay within the same interval number — it does not limit the allocation size. [6](#0-5) 

There is no per-call incentive ceiling independent of `amountAllocated`.

## Rejection Checks

**expected behavior checked:** The cap is a safety measure, but the design intent is clearly that each call processes one interval's worth of allocation (a small fraction of the pool). Draining 100% of the pool in one call (98.5% burned + 1.5% to caller) is not expected behavior.

**prior report checked:** No NatSpec, README, or known-issues documentation in the repository acknowledges this accumulation-cap interaction as intentional.

**README/NatSpec checked:** No NatSpec on `_calculateIntervalsETHBurn` or `swapETHForTorusAndBurn` documents this behavior.

**unsupported assumption checked:** The precondition (20 days of inactivity) is realistic — the protocol has no liveness enforcement, and the live state shows `totalETHBurn = 2,218,081,612,310,240,917 wei` (~2.22 ETH). The claim of "~99 ETH" in the question is incorrect — that figure is `ethUsedForBurns` (already spent), not `totalETHBurn`. The actual extractable incentive at current state is `2.22 ETH * 1.5% ≈ 0.033 ETH`, but `totalETHBurn` grows with each `distributeETHForBurning` call and could be substantially larger at the time of exploitation.
