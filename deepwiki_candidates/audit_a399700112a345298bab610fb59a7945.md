- `amountPerInterval = totalTitanXBurn * 200 / 10000 / 144`
- Cap triggers when `amountPerInterval * (1 + missedIntervals) ≥ totalTitanXBurn`

**attacker-controlled inputs:** `_deadline` (any valid future timestamp)

**call sequence:**

2. Call `swapTitanXForTorusAndBurn(_deadline)` as any EOA
3. `burnTitanXIntervalUpdate` → `_intervalUpdateTitanXForBurning()` → `_calculateIntervalsTitanXBurn()` sets `amountAllocated = totalTitanXBurn` (cap hit)

## Why Existing Checks Fail

- **`IntervalAlreadyBurned`** (line 231): only prevents double-claiming the *same* interval number; does not limit the size of `amountAllocated`. [6](#0-5) 
- **`nonReentrant`**: prevents reentrancy, irrelevant to accumulation size.
- **Cap `if (_totalAmountForInterval > totalTitanXBurn)`** (line 598): this is the mechanism that *enables* the attack — it caps at the full pool, not at a single interval's normal allocation. There is no secondary cap on the incentive amount itself. [7](#0-6) 
- **`DAILY_ALLOCATION_TITAN_X_BURNING` range [100, 1000]**: owner-settable but does not prevent accumulation over time; it only affects the per-interval base rate.
- No maximum incentive per call, no per-call incentive ceiling, no minimum call frequency enforcement.

## Rejection Checks

**expected behavior checked:** The protocol intends callers to be incentivized for gas costs on regular 10-minute intervals. Extracting 1.5% of the entire ~390e27 TitanX burn pool in a single call is not consistent with that design intent. The accumulation mechanism is meant to handle occasional missed intervals, not 50-day drains. [8](#0-7) 

**prior report checked:** No prior report or known-issues list found in the repository.

**README/NatSpec checked:** No NatSpec on `swapTitanXForTorusAndBurn` or `_calculateIntervalsTitanXBurn` documents intentional unbounded accumulation as a design choice. [9](#0-8) 

**unsupported assumption checked:** No oracle manipulation, no flash loan, no privileged role required. The attacker only needs to wait and call a public function.
