## Why Existing Checks Fail

`releaseFractalFunds()` has exactly one guard ŌĆö the time check ŌĆö and no `onlyOwner` or equivalent. [5](#0-4) 

The design intent is that fractal release happens as a side-effect inside `swapTitanXForTorusAndBurn` / `swapETHForTorusAndBurn` / `distributeTitanXForBurning` / `distributeETHForBurning` / `distributeTitanXForBuilding` / `distributeETHForBuilding` ŌĆö all of which call `releaseFractalFunds()` AFTER the interval allocation is already snapshotted. [6](#0-5) 

By calling `releaseFractalFunds()` standalone first, the attacker causes the interval update modifier to observe the post-release (inflated) `totalTitanXBurn`, yielding a larger `amountAllocated` and thus a larger incentive transfer. The `nonReentrant` guard on `swapTitanXForTorusAndBurn` does not prevent this two-transaction (or two-call) sequence. [7](#0-6) 

## Rejection Checks

**expected behavior checked:** The protocol clearly intends fractal release to be a side-effect of normal swap/distribute calls, not a standalone pre-call that inflates the current interval. The `public` visibility appears to be an oversight enabling internal calls, not an intentional permissionless trigger.

**prior report checked:** No NatSpec, README, or known-issues documentation in the repository describes this ordering as intentional or acceptable.

**README/NatSpec checked:** `releaseFractalFunds()` has no NatSpec. No documentation states it is callable by anyone at any time as a standalone operation.

**unsupported assumption checked:** No oracle manipulation, flash loan, or external dependency failure required. Pure on-chain state ordering with live-confirmed non-zero `fractalTitanX` and `fractalETH`.
