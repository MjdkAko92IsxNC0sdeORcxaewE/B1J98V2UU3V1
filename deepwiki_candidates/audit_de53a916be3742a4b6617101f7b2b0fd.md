## Attacker Path

**preconditions:**
- No stakers exist for days 2 through N (i.e., `totalShares[d] == 0` for all d in [2..N]). This is realistic at protocol launch or after a period of inactivity.
- `rewardPool[1] = INITIAL_DAILY_REWARD = 100_000e18` is set in the constructor. [4](#0-3) 
- Attacker has enough TitanX (or ETH) to pay the `createTorus` fee.

**attacker-controlled inputs:**
- `power` (any valid value, e.g. 1)
- `lengthInDays = 1` (minimum stake, minimizes cost while capturing full day-N+1 rewards)

**call sequence:**
1. Warp N days with zero protocol interactions (no `createTorus`, `stakeTorus`, `claim`, etc.)
2. Attacker calls `createTorus(power, 1)` on day N+1:
   - `_checkAndUpdateRewardPool()` runs with `currentDay = N+1`, `lastFilled = 1`
   - Loop `d = 2..N+1`: for each `d` in `[2..N]`, `totalShares[d] == 0` and `d != currentDay`, so:
     ```
     penaltiesInRewardPool[N+1] += rewardPool[d]   // line 707
     ```
   - After loop: `penaltiesInRewardPool[N+1]` = Σ `rewardPool[d]` for d=2..N
   - `_addShares(N+1, 1, shares)` → attacker is sole holder of `totalShares[N+1]`
3. Warp 1 day (to satisfy `block.timestamp >= stakePos.endTime`)
4. Attacker calls `claim(0)`:
   - `calculateStakingRewards` iterates over day N+1 only (stakingDays=1)
   - `totalShares[N+1] != 0`, so:
     ```
     reward += rewardPool[N+1] * shares / totalShares[N+1]          // = rewardPool[N+1]
     reward += penaltiesInRewardPool[N+1] * shares / totalShares[N+1] // = Σ rewardPool[2..N]
     ```
   - `torus.mint(msg.sender, principal + reward)` delivers all N-1 days of accumulated rewards

**Concrete magnitude (N=30):**
- `rewardPool[1] = 100,000e18`, `DAILY_REDUCTION_RATE = 8` BPS (0.08%)
- `penaltiesInRewardPool[31]` ≈ Σ rewardPool[2..30] ≈ 29 × ~98,800e18 ≈ **2,865,200 TORUS**
- Attacker receives ~30× the single-day reward entitlement for 1 day of staking cost [5](#0-4) 

## Why Existing Checks Fail

There are **no guards** preventing this:

1. **`_checkAndUpdateRewardPool` unconditionally rolls empty-day `rewardPool[d]` into `penaltiesInRewardPool[currentDay]`** for every past day with `totalShares[d] == 0`. There is no check that the current staker was present during those days, no cap, and no separate accounting for "unclaimed base rewards" vs "penalty funds." [6](#0-5) 

2. **`calculateStakingRewards` treats `penaltiesInRewardPool` identically to `rewardPool`** — both are distributed pro-rata to whoever holds shares on that day. The function has no concept of whether the `penaltiesInRewardPool` balance originated from penalties or from rolled-up empty-day base rewards. [2](#0-1) 

3. **`_addShares` only adds shares starting from `currentDay`**, so the attacker's shares are correctly zero for days 2..N — but this doesn't matter because `calculateStakingRewards` skips days where `totalShares[d] == 0` (line 728) and only pays out on day N+1, where the attacker holds 100% of shares. [7](#0-6) 

4. **`claim` has no eligibility check** tying the claimant's entitlement to the origin of `penaltiesInRewardPool`. It simply calls `calculateStakingRewards` and mints. [8](#0-7) 

## Rejection Checks

**expected behavior checked:** The pool is named `penaltiesInRewardPool`, and the NatSpec/comments describe it as receiving penalties from late/early claims. Rolling base `rewardPool` emissions from empty days into this pool and distributing them to the first staker is not documented as intended behavior. The invariant violation is real. [9](#0-8) 

**prior report checked:** No prior report found in this repository.

**README/NatSpec checked:** No NatSpec on `_checkAndUpdateRewardPool` describes the empty-day rollup as a feature. The comment on line 684 says only "fill in empty rewardPool values of previous days and the next day" — it does not mention redistributing those values to the current day's stakers. [10](#0-9) 

**unsupported assumption checked:** The assumption that N days of inactivity can occur is realistic — the protocol launched recently (`startTimeStamp: 1752170400`) and early-stage protocols routinely have periods with no stakers. No oracle failure or external dependency is required.
