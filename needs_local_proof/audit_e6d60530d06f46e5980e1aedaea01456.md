## Attacker Path

**preconditions:**
- Pool `titanXTorusPool` has `observationCardinality = 1` (default at creation, never expanded). With cardinality=1, `(observationIndex + 1) % 1 = 0`, so `getOldestObservationSecondsAgo` always reads `observations(0)` — the single slot, which is the most recently written observation, not the oldest in any meaningful sense.
- `liquidityAdded = true` (already satisfied per live context).
- A non-zero interval allocation exists for `swapTitanXForTorusAndBurn`.

**attacker-controlled inputs:**
- Block N: attacker executes a large buy of TORUS with TITAN_X on the pool, moving the spot tick significantly upward (TORUS more expensive in TITAN_X terms). No new observation is written in the same block as the previous observation.
- Block N+1: `getOldestObservationSecondsAgo` returns `block.timestamp_N+1 - block.timestamp_N` ≈ 12 seconds. `secondsAgo` collapses from 900 to ~12.
- The TWAP over 12 seconds = `(tickCumulative_current - tickCumulative_at_T_N) / 12` = the manipulated tick, because the only reference point is the observation written at T_N (before the manipulation swap).
- `adjustedTorusAmount = expectedTorusAmount * (100 - 10) / 100` is computed against the manipulated (lower) TORUS quote, setting a permissive `amountOutMinimum`.
- Attacker front-runs `swapTitanXForTorusAndBurn` in block N+1 to push price further, protocol swap executes at a worse rate within the 10% slippage band, attacker back-runs to extract profit.

**call sequence:**
1. Block N: `attacker.buyTorus(largeAmount)` on the TORUS/TITAN_X pool.
2. Block N+1 (attacker front-run): `attacker.buyTorus(additionalAmount)` to push price further.
3. Block N+1: `swapTitanXForTorusAndBurn(deadline)` — protocol swaps TITAN_X at manipulated price, `amountOutMinimum` is set against the manipulated TWAP.
4. Block N+1 (attacker back-run): `attacker.sellTorus()` — attacker profits from the price impact.

## Why Existing Checks Fail

The fallback `if (oldestObservation < secondsAgo) secondsAgo = oldestObservation` is the root cause. It is intended as a graceful degradation for a young pool, but with cardinality=1 it is a permanent condition: the single observation slot is always the most recent one, so `oldestObservation` is always the time since the last swap (seconds to minutes), never 900 seconds. [3](#0-2) 

`consult` requires `secondsAgo != 0` (reverts with "BP"), so the exact same-block case (secondsAgo=0) causes a revert rather than fund extraction. The exploitable window is the next block onward, where `secondsAgo` is small but non-zero (1–12 seconds), making the TWAP indistinguishable from spot. [4](#0-3) 

The `titanXToTorusSlippage = 10` (10%) gives the attacker a 10% sandwich window. No cardinality expansion is performed anywhere in the contract. [5](#0-4) 

## Rejection Checks

**expected behavior checked:** The code explicitly targets a 15-minute TWAP (`uint32 secondsAgo = 15 * 60`) and degrades gracefully. With cardinality=1, the degradation is permanent and total — this is not expected behavior, it is a design gap.

**prior report checked:** Not found in this repository.

**README/NatSpec checked:** No NatSpec or documentation acknowledges cardinality=1 as an accepted limitation or documents that the TWAP window may collapse to seconds.

**unsupported assumption checked:** The claim that `secondsAgo = 0` causes fund extraction is incorrect — it causes a revert. The actual extraction path requires `secondsAgo >= 1` (next block). The broader cardinality=1 TWAP collapse is the real issue and is source-confirmed.
