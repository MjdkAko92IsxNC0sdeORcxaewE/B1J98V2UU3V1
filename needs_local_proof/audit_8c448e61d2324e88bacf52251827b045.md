- `swapTitanXForTorusAndBurn`: lines 226–257 — unprivileged entry point

The critical fallback is at line 1128:

```solidity
if (oldestObservation < secondsAgo) secondsAgo = oldestObservation;
```

If `oldestObservation == 0`, `secondsAgo` becomes 0, and `OracleLibrary.consult` with `secondsAgo = 0` returns the current spot tick — not a TWAP at all. The `amountOutMinimum` then equals `spotQuote * 90%`, making a single-block sandwich viable with no sustained manipulation required. [1](#0-0) 

For the mature-pool case (full 15-minute TWAP), the slippage tolerance is 10% (`titanXToTorusSlippage = 10` per live state):

```solidity
uint256 adjustedTorusAmount = (expectedTorusAmount * (100 - titanXToTorusSlippage)) / 100;
``` [2](#0-1) 

## Attacker Path

**preconditions:**
- Either (A) the TitanX/TORUS pool's `oldestObservation` is 0 or very small (pool is new or observation cardinality was just expanded), OR (B) the pool has low liquidity and the attacker can sustain a price deviation for the full TWAP window (≤15 min).
- A pending `swapTitanXForTorusAndBurn` call is observable in the mempool (or the attacker triggers it themselves after manipulation).

**attacker-controlled inputs:**
- Timing of front-run/back-run transactions around the protocol swap.
- In case (A): a single-block flash-loan to move spot price.
- In case (B): sustained large trades over the TWAP window to shift the time-weighted average tick.

**call sequence:**
1. [Case A] Attacker flash-loans TitanX, sells TORUS into the pool to push TORUS price down (fewer TORUS per TitanX = deflated quote). Since `secondsAgo = 0`, `getTorusQuoteForTitanX` returns the manipulated spot price.
2. [Case B] Attacker moves pool price over ≤15 minutes, warps/waits for TWAP to reflect the deflated price.
3. Attacker (or anyone) calls `swapTitanXForTorusAndBurn`. `_swapTitanXForTorus` computes `amountOutMinimum = deflatedTWAP * 90%`. The actual swap executes at the manipulated price, receiving far fewer TORUS than fair value.
4. Attacker back-runs: sells pre-positioned TORUS back at the now-restored price, capturing the price impact of the protocol's swap.

## Why Existing Checks Fail

The only slippage guard is `amountOutMinimum = getTorusQuoteForTitanX(amountTitanX) * (100 - titanXToTorusSlippage) / 100`. [3](#0-2) 

This guard is only as strong as the TWAP. There is no secondary check against an independent price source, no circuit breaker, and no minimum observation age requirement. The `oldestObservation` fallback silently degrades the TWAP to a spot price without reverting or alerting, which is the root structural weakness. [4](#0-3) 

For the mature-pool case, the 10% slippage tolerance is the only bound on attacker profit per swap — if the TWAP is shifted by exactly 10%, the swap executes at 90% of fair value and does not revert.

## Rejection Checks

**expected behavior checked:** TWAP-based slippage is a deliberate design choice, but the `oldestObservation = 0` degenerate case (spot price used as TWAP) is not expected behavior — it silently removes the oracle's manipulation resistance.

**prior report checked:** No prior report found in the indexed codebase.

**README/NatSpec checked:** No NatSpec on `getTorusQuoteForTitanX` documents the `oldestObservation` fallback behavior or its implications.

**unsupported assumption checked:** The mature-pool attack (case B) requires sustained multi-block price manipulation, which is expensive and may not be profitable given the pool's actual liquidity depth — this must be verified empirically. The live state (`liquidityAdded: true`, protocol running since June 2025) suggests the pool has a full observation history, making case B unlikely to be profitable. Case A (oldestObservation = 0) is the stronger path but requires the pool to be in a degenerate state.
