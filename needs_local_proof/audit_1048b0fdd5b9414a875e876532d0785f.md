## Attacker Path

**preconditions:**
- Pool has few stored observations (e.g., at or near initialization, or after a cardinality increase). `getOldestObservationSecondsAgo` returns a value much less than 900 seconds (15 min). The pool was initialized with `increaseObservationCardinalityNext(100)` but observations only accumulate as swaps occur.
- `liquidityAddedBuild == true` (pool seeded).
- A non-zero `titanXIntervalsBuild` allocation exists for the current interval.

**attacker-controlled inputs:**
- Timing of the call (choose a block where `oldestObservation` is small).
- Front-run and back-run transactions in the same block.

**call sequence:**
1. Attacker front-runs: sells a large amount of Torus into the TORUS/TITAN_X pool, crashing the spot price (and the near-spot TWAP when `secondsAgo` is tiny).
2. Attacker calls `swapTitanXForTorusAndBuild(deadline)`. `getTorusQuoteForTitanX` consults the pool with `secondsAgo = oldestObservation` (e.g., 1â€“5 seconds), returning a near-spot, manipulated quote. `amountOutMinimum = manipulatedQuote * 90 / 100` â‰ˆ near-zero.
3. The Uniswap router executes the swap at the depressed price; the protocol receives far fewer Torus than fair value.
4. Attacker back-runs: buys Torus back cheaply, profiting the spread. The protocol's TitanX allocation is partially extracted.

## Why Existing Checks Fail

The only slippage guard is `titanXToTorusSlippage = 10` (10%), applied multiplicatively to the TWAP quote: [5](#0-4) 

If the TWAP reference itself is manipulated (because `oldestObservation` is small and the window degrades to near-spot), the 10% tolerance is applied to an already-wrong base. The guard does not enforce a minimum against any external price feed or time-weighted average over a guaranteed minimum window. There is no `require(secondsAgo >= MIN_TWAP_WINDOW)` check before using the oracle result. [6](#0-5) 

`consult` only requires `secondsAgo != 0`, not any meaningful minimum window.

## Rejection Checks

**expected behavior checked:** Using TWAP with a fallback to `oldestObservation` is a known pattern, but silently shrinking the window to 1 second without reverting or enforcing a minimum is not expected safe behavior â€” it defeats the purpose of the TWAP protection.

**prior report checked:** No prior report found in this repository.

**README/NatSpec checked:** No NatSpec on `getTorusQuoteForTitanX` documents the `oldestObservation` fallback as intentional or safe.

**unsupported assumption checked:** The attack does not require oracle failure or malicious token owner action â€” it only requires the pool to have a short observation history, which is guaranteed at deployment and possible after any cardinality increase.

```text
    function getTorusQuoteForTitanX(
        uint256 baseAmount
    ) public view returns (uint256 quote) {
        address poolAddress = torusToken.titanXTorusPool();
        uint32 secondsAgo = 15 * 60;
        uint32 oldestObservation = OracleLibrary.getOldestObservationSecondsAgo(
            poolAddress
        );
        if (oldestObservation < secondsAgo) secondsAgo = oldestObservation;
        (int24 arithmeticMeanTick, ) = OracleLibrary.consult(
            poolAddress,
            secondsAgo
        );
        uint160 sqrtPriceX96 = TickMath.getSqrtRatioAtTick(arithmeticMeanTick);
        quote = OracleLibrary.getQuoteForSqrtRatioX96(
            sqrtPriceX96,
            baseAmount,
            TITAN_X_ADDRESS,
            address(torusToken)
        );
    }
```
