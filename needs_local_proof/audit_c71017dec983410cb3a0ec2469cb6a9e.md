## Attacker Path

**preconditions:**
- `liquidityAdded == true` (confirmed live: `true`)
- `totalTitanXBurn > 0` (confirmed live: `390047518149437255354831991687`)
- `DAILY_ALLOCATION_TITAN_X_BURNING = 200` bps (2% per day, live confirmed)
- `INTERVALS_PER_DAY = 144` (10-min intervals)
- No one has called `swapTitanXForTorusAndBurn` for ≥ 7200 intervals (~50 days), so accumulated allocation hits the `totalTitanXBurn` cap

**attacker-controlled inputs:**
- `_deadline`: any valid future timestamp
- Timing: attacker simply waits; no capital required

**call sequence:**
1. Wait until `block.timestamp - lastBurnedTitanXIntervalStartTimestamp > N * INTERVAL_TIME` where N is large enough that `amountPerInterval * (N+1) >= totalTitanXBurn`
2. Call `swapTitanXForTorusAndBurn(deadline)`
3. `_intervalUpdateTitanXForBurning()` fires, sets `currInterval.amountAllocated = min(totalTitanXBurn, amountPerInterval * (missedIntervals+1))`
4. Incentive paid: `(currInterval.amountAllocated * 150) / 10000`
5. At cap: incentive = `(390047518149437255354831991687 * 150) / 10000 ≈ 5.85e27` TitanX in one tx

## Why Existing Checks Fail

The only guard against over-allocation is the cap `if (_totalAmountForInterval > totalTitanXBurn) { _totalAmountForInterval = uint128(totalTitanXBurn); }`. [5](#0-4) 

This cap does not limit the incentive — it merely prevents the allocated amount from exceeding the pool. The incentive is then computed as 1.5% of the full capped allocation: [6](#0-5) 

There is no per-call incentive ceiling, no check that the caller is the "rightful" interval executor, and no mechanism to limit the incentive to a single interval's proportional share. The `IntervalAlreadyBurned` guard only prevents double-spending the *same* interval number — it does not prevent a single call from sweeping all accumulated missed-interval allocation. [7](#0-6) 

With `DAILY_ALLOCATION_TITAN_X_BURNING = 200` bps and `INTERVALS_PER_DAY = 144`, the per-interval allocation is `totalTitanXBurn * 200 / 10000 / 144 ≈ 0.0139%` of the pool. After `144 * 50 = 7200` missed intervals (~50 days), the accumulated allocation hits the `totalTitanXBurn` cap, and the single-call incentive reaches its maximum of 1.5% of the entire burn pool.

## Rejection Checks

**expected behavior checked:** The accumulation of missed intervals is explicit in the code (`additionalAmount = amountPerInterval * missedIntervals`), but no NatSpec or documentation states that a single caller is *intended* to receive up to 1.5% of the entire pool. The design intent appears to be a per-interval incentive, not a bulk-sweep incentive. Not confirmed as expected behavior.

**prior report checked:** No prior report found in the indexed repository.

**README/NatSpec checked:** No NatSpec on `swapTitanXForTorusAndBurn` or `_calculateIntervalsTitanXBurn` documents the accumulation behavior or its incentive implications.

**unsupported assumption checked:** The attacker needs no capital, no privileged role, and no oracle manipulation. The only requirement is time elapsed — a realistic condition given the live `lastTitanXBurnIntervalNumber = 53087` and `lastBurnedTitanXIntervalStartTimestamp = 1784022600` showing the function is called regularly but could lapse.
